# 产品使用说明书

## 1. 产品概述

智能零售用户行为分析系统是一套面向零售运营、数据分析、客服和管理人员的内部 Web 应用，同时提供演示用户购物端。系统通过用户端产生真实购物行为，由后端采集并沉淀为行为事件，再在运营端展示核心指标、行为明细、转化漏斗、商品热度、用户画像、用户分群、预警、智能推荐、购买意向、报表和审计信息。

当前项目适合作为课程实训、MVP 演示和功能联调环境使用。系统重点展示完整业务闭环：

1. 用户在购物端搜索、浏览、收藏、加购、下单和支付。
2. 后端采集并去重行为事件。
3. 运营端刷新看板，查看指标、漏斗、商品排行和用户画像。
4. 有权限的角色处理预警、导出报表并查看审计日志。

## 2. 运行环境

### 2.1 技术栈

| 层级 | 技术 |
| --- | --- |
| 前端 | React、TypeScript、Vite |
| 后端 | FastAPI、Pydantic、SQLAlchemy |
| 数据库 | MySQL 8.0+ |
| 缓存 | Redis 7，可用于刷新状态、锁和缓存 |
| 迁移 | Alembic |
| 测试 | pytest、Playwright |

### 2.2 本地端口

| 服务 | 地址 |
| --- | --- |
| 前端 | `http://127.0.0.1:5173` |
| 后端 | `http://127.0.0.1:8000` |
| API 根路径 | `http://127.0.0.1:8000/api` |
| MySQL | `localhost:3306` |
| Redis | `localhost:6379` |

## 3. 启动系统

### 3.1 Docker Compose 启动

适合希望一次性启动 MySQL、Redis、后端和前端的场景。

```bash
docker compose up --build
```

启动完成后访问：

- 前端页面：`http://127.0.0.1:5173`
- 后端 API：`http://127.0.0.1:8000/api`

### 3.2 本地分别启动

后端：

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -e .[dev]
uvicorn api.app:app --reload --host 127.0.0.1 --port 8000
```

前端：

```bash
cd frontend
npm install
npm run dev
```

前端开发服务默认将 `/api` 请求代理到 `http://127.0.0.1:8000`。

### 3.3 环境变量

项目根目录提供 `.env.example`，常用配置如下：

| 变量 | 说明 | 示例 |
| --- | --- | --- |
| `DATABASE_URL` | MySQL 连接地址 | `mysql+pymysql://retail:2112@localhost:3306/retail_behavior` |
| `REDIS_URL` | Redis 连接地址 | `redis://localhost:6379/0` |
| `MASKING_DEFAULT` | 是否默认脱敏 | `true` |
| `EXPORT_STORAGE_PATH` | 报表导出目录 | `./var/exports` |
| `METRIC_FRESHNESS_MINUTES` | 指标新鲜度目标 | `5` |
| `ALERT_REFRESH_MINUTES` | 预警刷新目标 | `10` |

## 4. 登录与角色

打开前端后，在登录页输入演示账号登录。不同账号会进入不同工作台或购物端。

### 4.1 内部管理端账号

| 登录 ID | 角色 | 默认入口 | 权限说明 |
| --- | --- | --- | --- |
| `admin001` | 系统管理员 | 运营总览 | 全量查看、预警处理、报表导出、审计日志 |
| `ops001` | 运营经理 | 运营总览 | 运营分析、预警处理、报表导出 |
| `analyst001` | 数据分析师 | 转化漏斗 | 行为、漏斗、画像、推荐和报表分析 |
| `service001` | 客服主管 | 行为明细 | 查看行为路径、画像、预警，可处理预警 |
| `viewer001` | 只读查看员 | 运营总览 | 只读查看汇总、漏斗、商品热度和报表 |

### 4.2 用户购物端账号

| 登录 ID | 用户类型 | 行为特点 |
| --- | --- | --- |
| `user001` | 高价值运动用户 | 偏好鞋服、运动和数码商品，支付能力较高 |
| `user002` | 价格敏感家庭用户 | 偏好食品、母婴、家居和美妆，关注低价高销量 |
| `user003` | 潜力数码用户 | 频繁浏览数码和箱包，收藏加购较多 |

## 5. 用户购物端操作说明

用户购物端用于模拟真实用户行为，是运营端数据的主要来源。

### 5.1 搜索商品

1. 使用 `user001`、`user002` 或 `user003` 登录。
2. 在搜索框输入关键词，例如“跑鞋”“耳机”“水壶”。
3. 系统会记录 `search` 行为事件。
4. 商品列表根据关键词和分类展示匹配结果。

### 5.2 浏览和查看商品

1. 在商品列表中点击“查看”。
2. 系统打开商品详情面板。
3. 系统会记录 `browse` 和 `click` 行为事件。
4. 详情中展示品牌、类目、标签、评分、销量、库存、价格和推荐理由。

### 5.3 收藏商品

1. 在商品卡片或详情面板点击“收藏”。
2. 系统记录 `favorite` 行为事件。
3. 该行为会影响用户画像、兴趣偏好、购买意向和推荐结果。

### 5.4 加入购物车

1. 在商品卡片或详情面板点击“加购”或“加入购物车”。
2. 系统记录 `cart_add` 行为事件。
3. 该行为会进入转化漏斗的加购阶段。

### 5.5 下单并支付

1. 在商品卡片点击“立即购买”，或在详情面板点击“下单并支付”。
2. 系统依次记录 `order_submit` 和 `payment_success` 行为事件。
3. 运营端刷新后可在核心指标、转化漏斗、商品热度、销售额和报表中看到变化。

### 5.6 查看用户端推荐

用户端商品排序会结合用户画像种子、历史行为、价格偏好和商品销量进行推荐。用户产生更多浏览、收藏、加购和支付行为后，推荐结果会更贴合当前用户。

## 6. 运营管理端操作说明

内部账号登录后进入运营管理端。左侧菜单会根据角色权限自动显示可访问模块。

### 6.1 顶部筛选与刷新

运营端顶部提供时间范围、渠道、关键词和自动刷新能力。

- 时间范围：用于表达当前分析区间，例如“今日”。
- 渠道筛选：可按搜索、推荐、活动页、门店、直接访问筛选展示。
- 关键词筛选：可筛选商品、用户或搜索词相关行为。
- 自动刷新：开启后前端定时重新拉取核心指标、漏斗、行为、预警、报表和审计数据。

### 6.2 运营总览

入口角色：系统管理员、运营经理、数据分析师、只读查看员。

主要内容：

- 访问量
- 点击量
- 加购量
- 订单量
- 销售额
- 转化率
- 渠道来源分布
- 搜索关键词分布
- 商品热度摘要
- 预警摘要

使用建议：

1. 先确认顶部是否有接口错误提示。
2. 查看核心指标是否随用户端操作实时变化。
3. 结合渠道分布判断当前流量来源。
4. 结合预警摘要定位异常转化或行为数据问题。

### 6.3 行为明细

入口角色：系统管理员、运营经理、数据分析师、客服主管。

主要内容：

- 最近行为事件列表
- 用户或游客标识
- 商品 ID 和商品名称
- 行为类型
- 渠道
- 发生时间
- 幂等状态

使用步骤：

1. 使用用户端产生几条行为。
2. 切换到内部账号，例如 `ops001`。
3. 进入“行为明细”。
4. 通过渠道和关键词筛选定位行为。
5. 检查同一会话下是否能看到浏览、点击、加购、下单、支付链路。

### 6.4 转化漏斗

入口角色：系统管理员、运营经理、数据分析师、只读查看员。

主要内容：

- 浏览阶段
- 点击阶段
- 加购阶段
- 下单阶段
- 支付阶段
- 各阶段进入人数、转化率和流失率
- 最大流失阶段

使用建议：

1. 先通过用户端制造完整购买路径。
2. 刷新运营端漏斗。
3. 观察浏览到点击、点击到加购、加购到下单、下单到支付的转化。
4. 若某阶段流失高，可结合行为明细和商品热度继续分析。

### 6.5 商品热度

入口角色：系统管理员、运营经理、数据分析师、只读查看员。

主要内容：

- 商品名称
- 类目
- 浏览数
- 点击数
- 收藏数
- 加购数
- 购买数
- 转化率

使用建议：

1. 重点关注浏览高但购买低的商品。
2. 对比收藏、加购和支付之间的差异，判断价格、库存、活动或详情页是否需要优化。
3. 对转化率高的商品可增加推荐或活动曝光。

### 6.6 用户画像

入口角色：系统管理员、数据分析师、客服主管。

主要内容：

- 用户访问、点击、收藏、加购、下单、支付数量
- 支付金额和客单价
- 用户价值等级
- 购买意向等级
- 用户分群
- 浏览类目、收藏类目、购买类目和价格带分布

使用建议：

1. 对高价值用户查看其偏好类目和价格带。
2. 对潜力用户重点关注收藏和加购行为，制定召回或优惠策略。
3. 对客服场景，使用脱敏用户信息辅助跟进。

### 6.7 预警中心

入口角色：系统管理员、运营经理、客服主管。

主要内容：

- 低活跃提醒
- 流失风险预警
- 转化异常预警
- 行为数据异常
- 预警级别和处理状态

使用步骤：

1. 进入“预警中心”。
2. 查看触发原因和严重级别。
3. 有操作权限的角色点击“处理”。
4. 处理动作会记录到审计日志。

### 6.8 智能推荐

入口角色：系统管理员、运营经理、数据分析师。

主要内容：

- 推荐商品
- 推荐分数
- 推荐原因
- 推荐依据，包括浏览历史、购买历史、相似用户和用户画像

使用建议：

1. 先在用户端产生偏好行为。
2. 切换到推荐分析页面。
3. 查看推荐商品是否与用户偏好类目、价格带和历史行为匹配。
4. 将推荐原因用于运营解释和策略复盘。

### 6.9 报表统计

入口角色：系统管理员、运营经理、数据分析师、只读查看员。

报表类型：

- 用户行为明细
- 商品分析报表
- 转化漏斗报表
- 用户画像报表
- 用户分群报表
- 销售转化报表
- 运营效果报表

使用步骤：

1. 进入“报表统计”。
2. 查看当前角色可见报表。
3. 有导出权限的角色点击导出。
4. 导出成功后可下载文件。
5. 导出操作会写入审计日志。

权限说明：

- `admin001`、`ops001`、`analyst001` 可导出。
- `service001` 和 `viewer001` 无导出权限。
- 敏感字段默认脱敏。

### 6.10 审计日志

入口角色：系统管理员。

主要内容：

- 操作者
- 角色
- 操作动作
- 资源类型
- 资源 ID
- 操作结果
- 操作原因
- 记录时间

使用建议：

1. 导出报表后查看是否新增导出审计。
2. 处理预警后查看是否新增处理记录。
3. 发生权限拒绝时检查是否记录 `denied`。

## 7. 推荐演示流程

### 7.1 从零演示完整闭环

1. 使用 `admin001` 登录运营端。
2. 如需清空环境，调用 `/api/system/reset` 或在调试工具中触发系统重置。
3. 退出或切换到 `user001`。
4. 搜索“跑鞋”，查看商品详情，收藏，加购，立即购买。
5. 切换到 `ops001`。
6. 查看“运营总览”，确认访问量、点击量、加购量、订单量、销售额和转化率变化。
7. 查看“行为明细”，确认行为链路完整。
8. 查看“转化漏斗”，确认支付阶段有数据。
9. 查看“商品热度”，确认对应商品排行上升。
10. 查看“报表统计”，导出行为明细或漏斗报表。
11. 使用 `admin001` 查看“审计日志”，确认导出操作有记录。

### 7.2 角色权限演示

1. 使用 `viewer001` 登录。
2. 确认只能看到运营总览、转化漏斗、商品热度和报表。
3. 进入报表页，确认没有导出能力。
4. 使用 `service001` 登录。
5. 确认可查看行为明细、用户画像和预警，但不能导出报表。
6. 使用 `admin001` 登录。
7. 确认可查看审计日志和完整菜单。

### 7.3 用户画像与推荐演示

1. 使用 `user002` 搜索“水壶”“儿童”“低糖”等关键词。
2. 收藏或加购食品、母婴、家居商品。
3. 切换到 `analyst001`。
4. 查看用户画像和智能推荐。
5. 对比用户偏好类目、价格带和推荐理由。

## 8. 数据口径说明

### 8.1 行为事件口径

| 行为 | 说明 |
| --- | --- |
| 浏览 | 用户查看商品或页面 |
| 点击 | 用户点击商品或入口 |
| 搜索 | 用户输入关键词搜索 |
| 收藏 | 用户收藏商品 |
| 加购 | 用户加入购物车 |
| 下单 | 用户提交订单 |
| 支付成功 | 用户完成支付 |

### 8.2 核心指标口径

| 指标 | 说明 |
| --- | --- |
| 访问量 | `browse` 行为数量 |
| 点击量 | `click` 行为数量 |
| 加购量 | `cart_add` 行为数量 |
| 订单量 | `order_submit` 行为数量 |
| 销售额 | `payment_success` 事件关联商品价格和数量汇总 |
| 转化率 | 支付成功数 / 访问量 |

### 8.3 用户分群口径

| 分群 | 说明 |
| --- | --- |
| 高价值用户 | 支付金额高或支付次数多 |
| 潜力用户 | 收藏、加购较活跃但支付少 |
| 新用户 | 行为较少或刚开始访问 |
| 价格敏感用户 | 偏好低价格带商品 |
| 沉默用户 | 近期无明显行为 |
| 流失风险用户 | 活跃下降或长时间未转化 |

## 9. 常见问题

### 9.1 前端提示接口不可用

处理方式：

1. 确认后端服务已启动在 `http://127.0.0.1:8000`。
2. 确认前端开发服务已启动在 `http://127.0.0.1:5173`。
3. 确认 `frontend/vite.config.ts` 中 `/api` 代理指向后端。
4. 刷新页面后重新登录。

### 9.2 用户操作后运营端没有数据

处理方式：

1. 确认用户端操作后页面有“最近行为”或操作提示。
2. 切换到运营端后点击刷新，或开启自动刷新。
3. 确认后端没有重启丢失内存演示数据。
4. 若使用 MySQL 持久化，确认数据库连接和迁移已完成。

### 9.3 报表不能导出

处理方式：

1. 确认当前角色具备导出权限。
2. 使用 `admin001`、`ops001` 或 `analyst001` 重新登录验证。
3. 确认后端 `EXPORT_STORAGE_PATH` 指向可写目录。
4. 查看审计日志或后端日志定位权限拒绝原因。

### 9.4 登录 ID 不存在

处理方式：

1. 使用本文档列出的演示账号。
2. 注意登录 ID 全部为小写字母和数字。
3. 若浏览器保存了旧身份，清理 `localStorage.retail-auth` 后刷新。

## 10. 使用注意事项

1. 当前系统以演示和实训为主，生产环境需要补充正式账号体系、HTTPS、密码加密、细粒度 RBAC 和操作审批。
2. 行为采集依赖 `eventId` 幂等，测试时不要复用同一个事件 ID 验证新增数据。
3. 敏感信息默认脱敏，报表导出需要明确业务原因并记录审计。
4. 系统重置会清空演示数据，只建议在开发和演示开始前使用。
5. 指标、漏斗和推荐结果依赖已有行为事件，若没有用户端行为，页面会展示空状态或较少数据。
