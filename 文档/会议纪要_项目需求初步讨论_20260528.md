## 一、会议目的

讨论并明确“智能零售用户行为分析系统”项目的核心需求、功能范围、技术路线、非功能约束以及项目推进计划，为后续需求规格说明书、系统设计文档、数据库设计和项目开发奠定基础。

---

## 二、项目背景

随着零售业务数字化转型深入，企业需要对用户在平台上的行为数据进行全面采集、分析和可视化展示。当前业务中存在行为数据分散、统计口径不统一、用户路径难追踪、运营分析效率低等问题。

本项目拟建设一套智能零售用户行为分析系统，实现用户浏览、点击、搜索、收藏、加购、下单、支付等全链路行为数据采集与分析，为运营决策、用户画像、商品优化、转化提升和智能推荐提供数据支撑。

---

## 三、技术路线讨论结论

经讨论，项目统一采用以下技术路线：

| 层级 | 技术选型 | 说明 |
|---|---|---|
| 前端 | React + TypeScript + Vite | 适合多角色工作台、购物端和复杂数据看板 |
| 后端 | FastAPI + Pydantic | 开发效率高，自动生成 Swagger/OpenAPI，适合数据分析和推荐算法 |
| 数据库 | MySQL 8.x | 作为系统核心持久化数据库，存储用户、商品、行为、订单、支付、日志等数据 |
| ORM/迁移 | SQLAlchemy + Alembic | 管理 MySQL 表结构、实体映射和数据库迁移 |
| 图表 | ECharts / 前端图表组件 | 支持漏斗图、饼图、柱状图、趋势图和排行图 |
| 缓存 | Redis（可选增强） | 用于热点指标缓存、刷新状态、任务锁等 |
| 测试 | pytest + Playwright | 后端接口测试、业务逻辑测试和前端端到端流程验证 |

## 四、功能需求讨论

### 4.1 用户行为数据采集

系统需要准确采集并统计以下 7 类用户行为数据，不得出现重复统计或漏统计：

- 浏览行为：商品详情页浏览、商品列表浏览
- 点击行为：商品点击、推荐位点击、广告位点击
- 搜索行为：关键词搜索、筛选搜索
- 收藏行为：收藏商品
- 加购行为：加入购物车
- 下单行为：提交订单
- 支付行为：完成支付

行为数据需记录用户 ID、会话 ID、行为类型、目标对象 ID、商品 ID、订单 ID、支付 ID、来源渠道、时间戳和扩展信息。

### 4.2 用户行为路径追踪

系统需支持用户从访问到支付的全链路行为追踪：

```text
访问 / 浏览 -> 点击 -> 收藏 / 加购 -> 下单 -> 支付
```

每一步行为的时间、来源渠道、关联商品、订单和支付信息均应可追溯。

### 4.3 多维筛选分析

系统需支持以下维度组合筛选：

- 时间维度：日、周、月、自定义时间区间
- 商品维度：品类、品牌、SKU、价格区间
- 用户维度：用户 ID、会员等级、用户分群、购买意向
- 渠道维度：搜索、推荐、活动页、门店、直接访问
- 行为维度：浏览、点击、搜索、收藏、加购、下单、支付

### 4.4 实时数据看板

核心指标需支持 5 秒级刷新：

- 访问量
- 点击量
- 收藏量
- 加购量
- 订单量
- 支付量
- 销售额
- 转化率

### 4.5 用户画像与分群

用户画像需基于 MySQL 中的长期行为数据计算：

- 基础属性画像
- 消费能力画像
- 兴趣偏好画像
- 活跃度画像
- 用户价值等级
- 购买意向等级

用户分群包括：

- 高价值用户
- 潜力用户
- 新用户
- 价格敏感用户
- 沉默用户
- 流失风险用户

### 4.6 智能推荐

推荐模块需结合以下因素计算推荐分：

- 用户浏览偏好
- 用户收藏偏好
- 用户加购偏好
- 用户购买记录
- 商品热度
- 价位匹配度
- 未购买新鲜度

推荐结果需展示商品、推荐分和推荐原因。

### 4.7 权限管理

系统管理端规划 5 类角色：

- 系统管理员：全量数据、权限配置、审计日志、系统重置
- 运营经理：运营看板、行为分析、漏斗、商品热度、预警、报表
- 数据分析师：指标分析、画像、分群、推荐和报表
- 客服主管：用户路径、画像和风险提醒
- 只读查看员：脱敏汇总数据查看

前端按角色显示菜单，后端按角色校验接口权限。

### 4.8 异常数据检测

系统需对指标进行合理性校验：

- 浏览量、点击量、订单数、销售额不能为负数
- 转化率、留存率、流失率应在 0%-100%
- 出现异常时生成预警记录

## 五、非功能需求讨论

### 5.1 数据准确性

- 行为事件必须具备唯一事件 ID。
- 支持幂等去重。
- 支持批量上报。
- 采集失败需要记录日志。

### 5.2 数据持久化

- 用户、商品、行为、订单、支付、日志必须存入 MySQL。
- 前端不作为业务数据源。
- 后端分析结果应基于 MySQL 数据计算。

### 5.3 易用性

- 界面布局清晰。
- 图表直观。
- 支持筛选和下钻。
- 管理端和用户端入口清晰分离。

### 5.4 安全与审计

- 敏感信息默认脱敏展示。
- 查询、导出、重置、预警处理等关键操作记录操作日志。

## 八、决议事项

1. 项目继续命名为“智能零售用户行为分析系统”。
2. 技术栈统一为 FastAPI + React + TypeScript + Vite + MySQL。
3. MySQL 作为核心业务数据库，承载用户、商品、行为、订单、支付、画像和审计等持久化数据。
4. Redis 暂定为可选增强，不作为第一阶段必需项。
5. 用户端至少支持 3 个用户，管理端支持 5 个内部角色。
6. 画像、推荐、漏斗、指标统计均由后端基于 MySQL 数据计算。
7. 文档和项目计划均按本技术路线保持一致。
